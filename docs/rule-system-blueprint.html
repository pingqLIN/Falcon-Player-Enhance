Falcon Rule System vNext

不是再加更多規則，而是把規則真相來源重新統一。

這份圖面把現況的 DNR / MAIN world / DOM heuristic / CSS hide / AI advisory / user custom rules 重新思考了一次。 結論很直接：最佳方案不是擴張更多 blocklist，而是建立單一規則供應鏈，清楚分隔 baseline、runtime、policy overlay、review queue。

設計立場

Problem Heatmap

現況最大的不是規則不夠，而是規則分裂。

同一類風險同時存在於 JSON、background constants、content heuristics、AI seeds 與 storage custom rules， 導致補丁很快，但長期 drift、重複處理與回溯成本持續累積。
Truth Split

哪裡才算正式規則？

filter-rules.json、background.js、inject-blocker.js、cosmetic-filter.js 都持有自己的 truth fragment。

Duplicate Action

同一元素可能被重複處理

同一個 overlay 可能先被 CSS 隱藏，再被 DOM remover 處理，最後又被 MAIN world guard 攔一次。

AI Boundary

AI seeds 與正式規則容易混淆

ad-list.json 結構很像正式規則，但目前實際角色更接近 vocabulary 與 knowledge seeds。

Rollback Cost

出了誤判，難以快速回溯

因為 runtime behavior 不只一條鏈，誤刪與誤攔截往往無法快速回答是誰造成的。

Thought Experiments

四個腦內思想實驗後，最佳解不是「更多」，而是「更乾淨」。

這裡不是只評估效益，也把容易在實作時被忽略的副作用一起拉進來：性能、誤判、回退、資料污染與維護成本。
Experiment 1

保留現況多來源模型

補丁速度最快，但規則真相來源會繼續裂開。每多一個快速 patch，後面就多一份 drift 與回歸風險。

結論：適合救火，不適合當長期系統。
Experiment 2

把 ad-list.json 直接升格為唯一 blocklist

看起來統一，但實際上把 domain / selector / token / pattern 混成同一條執行管線，誤判模型會失真。

結論：可作 vocabulary，不該直接作唯一執行規則集。
Experiment 3

允許 AI 自動 durable mutation

學習速度確實最快，但 selector 污染與 false positive 也會一起加速，rollback 成本會遠高於現在的 repo 邊界。

結論：在目前架構能力下不值得啟用。
Experiment 4

把 enhanced chain 全域下放到所有站

陌生站首訪保護更早，但 document_start 注入、MAIN world 相容性與誤攔截風險也會同步上升。

結論：應改成風險觸發升級，不應全域最強注入。
New Architecture

新版規則體系：四層模型，單一規則供應鏈。

正式規則、執行期防護、AI 覆蓋與待審核規則必須拆開，才能同時保住播放器保護能力與長期可維護性。
左側是新的 canonical input board，右側是四層規則鏈。每一層只做一件事：baseline 定義正式底座，runtime guards 負責可逆防禦，policy overlay 負責可逆參數調整，review queue 接住可學習但尚未正式採用的規則。

Canonical Input Board

Baseline data
DNR request rules, high-confidence selectors, site behavior profiles
Runtime telemetry
popup, redirect, overlay_removed, suspicious_dom_churn
Policy overlay
blocking level, popupStrictMode, guardExternalNavigation, extraBlockedDomains
Review inputs
hiddenElements, AI candidates, teach mode output
1
Baseline Rules

正式底座，只放高信心、低誤判規則

承接 DNR request/domain rules、穩定 selector hide、站點 profile。
必須可 build、可測試、可追溯來源。
禁止
不直接吸收 AI candidate 與 user custom rules
2
Runtime Guards

只做執行期防禦，不再維護第二份長期 truth

處理 popup interception、overlay cleanup、fake video、player-adjacent traps。
只讀 baseline 與 policy overlay，不自行擴張正式規則。
典型模組
inject-blocker / merged overlay-remover chain
3
Policy Overlay

AI 與 blocking level 只能覆蓋參數，不能寫正式規則

允許 reversible actions：popupStrictMode、guardExternalNavigation、overlayScanMs、extraBlockedDomains。
仍維持 policy gate 與 host-scoped advisory。
邊界
不做 durable mutation
4
Review Queue

把可學習資料和正式底座拆開

接住 hiddenElements、generated candidates、teach mode 樣本。
經 review、驗證、回滾流程後，才允許升級為 baseline。
保護作用
避免 false positive 直接污染正式規則
Canonical Sources

規則來源重新收斂成三個正式入口。

目的不是把一切塞到單一 JSON，而是建立三種清楚的 source of truth：正式底座、執行期配置、待審核資料。
Canonical Baseline

rules/baseline-rules.*

承接高信心 request/domain rules、穩定 selectors 與可版本化的正式底座。

取代現行 filter-rules.json 的單一格式限制
吸收高信心 domain/token，不再散落在多處常數
可輸出 DNR、runtime config、selector bundles
Behavior Profile

rules/site-profiles.*

由 domain registry 升級成最小行為 schema，描述 enhanced chain、相容模式與站點行為。

站點是否啟用 enhanced chain
哪些 guard 需放寬或加嚴
哪些 iframe / player family 被視為安全媒體來源
Review State

storage review state

接住可學習、可自訂、尚未正式採用的規則，預設不是 baseline。

hiddenElements 與 reviewed selectors 分層
AI generated candidates 與 teach mode output
accepted / rejected / rollback log
Execution Order

新版仍保留快慢分層，但權責更清楚。

執行順序不需要被推翻；真正需要改的是每個階段到底該讀什麼規則、能不能自己擴張規則。
01

Background init

載入 baseline、site profiles、blocking level、AI state，決定當前 tab 是否升級到 enhanced chain。

02

DNR first

高信心 request/domain rules 先擋掉最不需要進頁面的流量與已知 trap host。

03

document_start guards

MAIN world 與 early isolated guards 只處理必須搶先攔截的 popup、redirect、anti-adblock 與已知 safe profile。

04

document_idle cleanup

player detector、merged overlay chain、fake video 與 enhancer 接手做播放器周邊的精細清理。

05

Policy overlay

AI 與 blocking level 只能調 runtime knobs，不能把學到的規則直接升格成正式底座。

Migration Route

不是大爆炸重寫，而是五段式遷移。

每一步都應該能獨立驗證與回退，避免在 MAIN world 與 DOM heuristic 上一次引入太多未知數。
Phase 1

定義 schema

先建 baseline 與 site profile schema，完成 source-of-truth mapping，不先碰核心行為。

Phase 2

合併 popup / overlay chain

把 anti-popup 與 overlay-remover 收斂為同一條 DOM heuristic 管線。

Phase 3

抽離 MAIN world truth

讓 inject-blocker 不再長期持有第二份 domain/block token truth。

Phase 4

限縮 cosmetic filter

只消費 baseline selectors 與 reviewed custom selectors，不再兼任動態 heuristic。

Phase 5

建立 review pipeline

讓 AI candidates 與 user custom rules 都經 review/rollback 流程，再決定是否升級為 baseline。

最終判斷

最佳方式不是新增一套規則，而是建立單一規則供應鏈。 只有把 baseline、runtime guards、 policy overlay、review queue 的邊界切開， 才能同時保住播放器保護能力、AI 可逆強化能力、使用者自訂彈性與長期維護性。
