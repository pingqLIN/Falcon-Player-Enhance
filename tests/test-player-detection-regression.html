    Case 3: Preview 縮圖影片

    等待掃描

    這是小尺寸、靜音、自動播放的預覽縮圖，不應進入候選名單。

    Case 4: 廣告 iframe

    等待掃描

    容器與來源都帶有 ad / banner 訊號，應被排除。

    預期：不應被偵測，不應進 popup list。

    Case 5: Trap 影片

    等待掃描

    中等尺寸，但為 muted + autoplay + loop + no controls 的陷阱型態。

    預期：不應被視為主要播放器。

    Case 6: 小尺寸但真實可播放影片

    等待掃描

    尺寸接近邊界，但有 controls 與真實來源，應保留為有效播放器。

    預期：應被偵測，驗證小尺寸真影片不會被誤殺。

    Case 7: 延遲注入來源的播放器

    等待掃描

    初始只有空白 video controls 佔位元素，稍後才插入 source。應在來源出現後才被偵測。

    預期：延遲注入 source 後應被偵測，驗證 attribute-driven redetect。

    Case 8: 延遲設定 src 的 generic iframe

    等待掃描

    這個 iframe 先不帶 src，稍後才補上含 player/embed 的來源，應在更新後被 detector 接住。

    預期：來源更新後應被偵測為 generic playable iframe。

    Case 9: 自訂 shell 內含真實 media child

    等待掃描

    容器本身長得像 player shell，內部才有真實影片。這類 custom container 不應被新的 eligibility 規則誤拒。

    預期：應被偵測，驗證 custom container + playable child 邊界。

    Case 11: 隱藏 popup button 不得阻擋互動

    等待掃描

    模擬滿頁影片卡片場景。當 popup button 處於隱藏狀態時，不應擋住底下的 input / button 點擊。

    預期：播放器仍可被偵測，但隱藏中的 popup button 不應攔截 input / button。
